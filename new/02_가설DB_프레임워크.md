# 02. 가설 DB 프레임워크

---

## 설계 원칙

### 핵심 규칙 (4가지)

1. **시즌성 가설 DB 제외** — 시즌 소재는 변수 통제가 불가하므로 가설 DB와 분리
2. **단일 고정 대조군** — 모든 테스트에서 동일한 대조군 소재 사용
3. **1회 1변수 원칙** — 한 번의 테스트에서 한 가지 변수만 변경
4. **카테고리 순서 테스트** — Hook/오프닝 → 크리에이티브 스타일 순으로 진행

---

## 가설 카테고리 구성

| 카테고리 | 설명 |
|----------|------|
| Hook / 오프닝 | 영상 초반 0~2초 요소 |
| 크리에이티브 스타일 | 영상 전체 표현 방식 |
| 바디 | 중반부 콘텐츠 구성 |
| CTA | 행동 유도 문구/시각 요소 |
| 포맷 | 가로/세로, 길이, 정적/동적 |

> **현황**: 5개 카테고리, 총 17개 가설 수록 (DUC_UA_가설DB.xlsx)

---

## 첫 번째 재검증 가설

**가설**: 보상 장면(코인/잭팟 폭발 또는 당첨 금액 텍스트)을 영상 초반 0~2초 이내에 노출하면, 대조군 대비 IPM이 높다.

- **카테고리**: Hook / 오프닝
- **변경 변수**: 초반 0~2초 요소만 변경
- **고정 요소**: 바디, CTA 동일 유지
- **검증 방향**: Exploitation (슬롯 소재 요소 최적화) 우선

---

## 동시 vs. 순차 A/B 테스트

- 결정 보류 → **마케팅 전문가 합류 후 결정** 예정
- 3/25(수) 미팅에서 Phase 1 대조군 설정 + 테스트 세팅 확정 예정

---

## 전략적 방향 (2026년 3월 회의 결정)

| 구분 | 내용 |
|------|------|
| **우선순위** | Exploitation (슬롯 소재 요소 최적화) |
| **이후 단계** | Exploration (콘텐츠 유형 변경) |
| **훅 테스트 정의** | 초반 몇 초 요소만 변경, 바디/CTA 고정 |

---

## 주의사항

- 개별 기획자가 독립적으로 새 가설을 생성하는 방식 지양
- 기존 TF 테스트 데이터를 플레이북 Phase 1(Hook)에 재활용하는 방안 검토 중
- 가설 ID 없이는 성과-가설 자동 매칭 불가 → 네이밍 컨벤션 변경 필수
